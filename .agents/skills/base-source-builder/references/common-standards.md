# Common Standards

Apply these rules to every base source generated by this skill.

## 1. Naming Conventions

### Root Project Name

- Use `kebab-case` for the repository or root folder name.
- Keep it short and descriptive.
- Examples:
  - `order-service`
  - `inventory-api`
  - `admin-portal`

### Directory Names

- Use `kebab-case` for general directories.
- Use framework-native conventions when a stack requires a special pattern.
- Do not use spaces, uppercase letters, or vague names like `misc`, `common-stuff`, `temp-final`.

### File Names

- Config and documentation files keep ecosystem-native names.
- Java source files use `PascalCase.java` and class names must match the file name exactly.
- Python modules use `snake_case.py`.
- Vue single-file components use `PascalCase.vue`.
- Utility TypeScript files in Vue projects use `kebab-case.ts` unless a framework convention requires another name.

## 2. Required Cross-Project Files

Create these when they make sense for the chosen stack:

- `.editorconfig`
- `.gitignore`
- a dependency manifest
- a main entry point
- at least one test location

## 3. Formatting Baseline

Every scaffold must ship with formatting rules preconfigured:

- Line endings: `LF`
- Final newline: required
- Indentation:
  - Java: 4 spaces
  - Python: 4 spaces
  - VueJS, JSON, YAML, Markdown: 2 spaces
- UTF-8 encoding
- Remove trailing whitespace

Use `.editorconfig` to enforce the shared baseline.

## 4. Structure Rules

- Separate source code from tests.
- Separate framework bootstrapping from business logic.
- Keep config files at the project root unless the ecosystem requires otherwise.
- Prefer one obvious location for each concern:
  - config
  - source
  - tests
  - static assets

## 5. Default Project Quality Bar

- The scaffold must be runnable after dependency install.
- The scaffold must be format-ready on day one.
- The scaffold must avoid placeholder junk files unless the toolchain requires them.
- Choose the smallest standard layout that can grow cleanly.
